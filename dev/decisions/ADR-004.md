
# ADR-004: Vector store inicial para RAG

## Fecha
2025-12-04

## Estado
Aceptada

## Contexto
El agente de MolSys-AI necesita acceder a documentación y código mediante RAG:
- documentación Sphinx de MolSysMT, MolSysViewer, TopoMT,
- docstrings,
- ejemplos de uso.

Es necesario elegir un backend de almacenamiento vectorial para el MVP.

## Decisión
Usar **FAISS local** como vector store inicial.

## Opciones consideradas

### 1. FAISS local (elegida)
**Pros:**
- Muy rápido en CPU/GPU.
- No requiere servicios externos.
- Fácil de empaquetar y distribuir.
- Ideal para índices pequeños/medianos en un solo nodo.

**Contras:**
- No distribuido ni multi-nodo de forma nativa.

### 2. Chroma
**Pros:**
- API simple y amigable.
- Persistencia integrada.

**Contras:**
- Requiere servidor dedicado o proceso adicional.
- Más pesado para un MVP.

### 3. Weaviate / Milvus / Vespa
**Pros:**
- Alto rendimiento y escalabilidad.
- Características avanzadas de búsqueda.

**Contras:**
- Overkill para el tamaño inicial del proyecto.
- Añade complejidad operativa importante.

## Justificación
FAISS permite avanzar rápido sin añadir complejidad de infraestructura.  
El diseño del módulo `rag/` abstraerá la interfaz para poder cambiar el backend más adelante sin impactar al agente.

## Consecuencias
- Implementación simple y portable.
- Sustituible fácilmente por otro backend si el volumen de datos crece.

## Notas futuras
- Revisar la elección cuando el índice crezca significativamente o se necesite replicación/distribución.
